# setup-dod

leadcraft プラグインの **Definition of Done (DoD)** リストを対話的に登録・管理するスキル。

DoD は「個別 Story の受け入れ基準（Acceptance Criteria）とは別に、すべての Story が満たすべき共通の完了条件」である。テスト・レビュー・ドキュメント更新のように、毎 Story 同じことを書きたい項目を一元管理することで、書き漏れと表現揺れを防ぐ。

## 役割と他スキルとの契約

本スキルが保存した DoD は **契約** として次のスキルから参照される。本スキル単体では何も出力しないが、他スキルに渡る「共有された前提」を構築する。

| スキル | 利用方法 |
|--------|----------|
| `compose-stories` | Story 作成直後に `add_comment` 操作（抽象契約）で DoD をチェックリスト形式のフォローコメントとして付与する |
| `compose-hotfix` | Hotfix Story 作成直後に同じ仕組みで DoD コメントを付与する（緊急対応 Story にもチーム共通の完了条件は適用される） |
| `compose-epic`（モード B: Story 抽出） | 抽出した Story を `compose-stories` 経由で作成する際、上記コメント付与が連動して動く |

`compose-objective` / `compose-epic`（モード A）は対象外（Epic / Objective は完了概念がプロダクトロードマップ的で、Story のチェックリストとは粒度が違うため）。

`add_comment` の具体的な動作は `tracker.provider` によって異なる（`local`: Story md にフォローセクションを追記 / `github`: Issue にコメント投稿）。スキル本文は抽象操作名で記述し、具体手順は `references/backends/<provider>.md` のアダプタが提供する。

## データモデル

DoD は `.claude/leadcraft.md` の YAML フロントマターに、シンプルな **文字列リスト** として保存する。

```yaml
dod:
  - "単体テストを追加し、CI（lint / test / build）がすべて green である"
  - "レビューを 1 名以上から取得し、指摘事項を解決またはチケット化している"
  - "関連ドキュメント（README / API 仕様 / 操作手順）を更新している"
```

設計判断:
- 各項目は **文末まで完結した状態表現**（「〜である」「〜している」）にする。チェックリストにしたとき、人間が「これは満たした」と判定しやすい
- フラットなリストにとどめる。Epic / ラベル別の出し分けは現状サポートしない（必要になったらこの SKILL を拡張する）
- 並び順 = チェックリストの表示順。優先度の高い項目を上に置く

## 実行手順

### 1. 既存設定の確認

`.claude/leadcraft.md` の有無を確認する。

- 存在する場合: 現在の `dod:` セクションを読み込み、内容を箇条書きでユーザーに提示する。`dod:` キー自体が無いなら「未設定」として扱う。
- 存在しない場合: `${CLAUDE_PLUGIN_ROOT}/skills/setup-baseline/templates/leadcraft.md` を雛形にコピーする（`.claude/` ディレクトリが無ければ `mkdir -p` で作成する旨をユーザーに確認）。

### 2. サブモードの決定

引数で次のいずれかが渡されているならそれに従う。未指定なら `AskUserQuestion` で選ばせる。

| サブモード | 動作 |
|-----------|------|
| `add` | 新しい項目を末尾に追加。続けて何項目でも追加可 |
| `edit` | 既存項目を 1 件選んで書き換える |
| `remove` | 既存項目を 1 件選んで削除する |
| `reorder` | 既存項目の表示順を入れ替える |
| `list` | 現在の DoD を表示するだけ。設定は変更しない |
| `reset` | 既存の DoD を捨て、最低限の既定セット（テスト / レビュー / ドキュメントの 3 項目）で初期化 |

引数にテキストや md パスが渡された場合は、内容を箇条書きとして解釈して **bulk add** モードに入る（ユーザーに「以下を追加する」と提示してから確定）。

### 3. 項目の編集

追加・編集・リセットの場合、ユーザーから DoD 項目の文字列を引き出す。次の観点でレビューする。

- **状態表現か**: 「〜する」のような命令形ではなく「〜である / 〜している」のような状態表現になっているか。命令形のままでも保存はするが、よりよい言い換えがあれば提案する
- **検証可能か**: 「品質が高い」のような曖昧な表現を避け、具体的な検証手段（CI / レビュー / ドキュメント差分など）に言及させる
- **重複していないか**: 既存項目と内容が同義であれば、ユーザーに統合 / 分割を確認する

最低限の既定セット（`reset` 時 / 完全な空からスタートする時に初期提案）:

```yaml
dod:
  - "単体テストを追加し、CI（lint / test / build）がすべて green である"
  - "レビューを 1 名以上から取得し、指摘事項を解決またはチケット化している"
  - "関連ドキュメント（README / API 仕様 / 操作手順）を更新している"
```

### 4. 設定ファイルへの保存

`.claude/leadcraft.md` の `dod:` セクションのみを `Edit` で差し替える。

- 他のキー（`baseline` / `output` / `tracker` / `okf` / `issue` 等）は **絶対に消さない**
- `dod:` キー自体が無い既存設定ファイルには、適切な位置（`baseline:` の直後を推奨）に新規追加する
- YAML の文字列はダブルクォートで囲み、コロン・ハイフン・引用符の混入があっても壊れないようにする
- 全体置換が難しい場合は、`Read` で現在の内容を取得し、`dod:` ブロックの開始行〜終了行を特定して `Edit` で差し替える

### 5. 結果の表示と次のステップ

保存後、最新の DoD 全体をユーザーに見やすく提示する（番号付きの箇条書きを推奨）。
状況に応じて次のスキルを案内する。

- これから Story を作る場合: `/compose-stories`
- 既存 Epic を Story に分割したい場合: `/compose-epic <epic-readme-path>`
- 既存 Story への遡及反映が必要な場合: `tracker.provider` に応じた手順を案内する
  - `local`: Story md に DoD セクションを手動追記するか、一括スクリプトを別途用意する
  - `github`: `add_comment` 操作（抽象契約）で既存 Issue にコメントを追加する。具体手順は `references/backends/github.md` を参照

## 設定ファイルが無いときの初期化フロー

1. `Read` で `.claude/leadcraft.md` を確認 → 存在しない
2. `mkdir -p .claude` を実行（事前にユーザー確認）
3. `${CLAUDE_PLUGIN_ROOT}/skills/setup-baseline/templates/leadcraft.md` を読み、内容をそのままコピーして書き出す
4. その後、`dod:` セクションを 3. の既定セットで埋める
5. ユーザーに「baseline / tracker などの他項目も未設定なので、必要に応じて `setup-baseline` などを実行してほしい」と案内

## 注意事項

- 本スキルはトラッカー（Issue / Story ファイル）への書き込みを行わない。設定ファイルの編集に閉じる
- `dod:` セクションを **空リスト** にすることも許可する（DoD を一旦オフにしたい運用のため）。空のときは他スキルは DoD セクション / コメントを生成しない
- 過去の Story への遡及更新は本スキルの責務外。チームの運用判断で、必要なら一括スクリプトを別途用意する
- `.claude/leadcraft.md` をチームで共有する場合は、`tracker.github.*` などの個人・環境固有の値を含まないことを確認してからコミットする。DoD はチーム全員で揃えたい項目のため、共有設定ファイルの `dod:` セクションを更新してコミットし、新規メンバーがそこから取り込む運用を推奨する
- 1 項目は概ね 1 行で書く。長くなりそうなら 2 つに分けるか、補足は Story 個別の Acceptance Criteria 側で扱う
